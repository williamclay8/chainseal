# Chainseal v0.2.0 Proof Release

Positioning:

```text
Chainseal now has a real local trust loop.
```

This release proves the local path before any hosted surface: screen a candidate, verify source refs, store an allowed memory with an explicit receipt ledger, recall it as a lead, audit the ledger, and package the proof surface.

## Included

- Candidate schema.
- Receipt schema.
- Reusable library API.
- `chainseal store`.
- `chainseal recall`.
- `chainseal audit`.
- Expanded canaries.
- Threat model.
- Proof playbook.
- Repo entry packet.
- Public project spine.

## What Chainseal Blocks

The v0.2.0 canary suite proves Chainseal blocks:

- secret-like memory content;
- raw transcript-shaped memory content;
- unsupported or source-free memory claims;
- instruction-injection-shaped memory content;
- obfuscated instruction-injection-shaped memory content;
- missing source files;
- unscoped mutation-style actions that require review.

## Exact Commands

Run the release proof locally:

```bash
npm test
npm run pack:dry
git diff --check
```

Run only the canary suite:

```bash
skills/chainseal/scripts/chainseal-canary.sh .
```

Print schemas:

```bash
chainseal schema candidate
chainseal schema receipt
```

Gate, store, recall, and audit:

```bash
chainseal gate candidate.json --project .
chainseal store candidate.json --ledger /tmp/chainseal-receipts.jsonl --project .
chainseal recall "source-backed memory" --ledger /tmp/chainseal-receipts.jsonl --project .
chainseal audit --ledger /tmp/chainseal-receipts.jsonl --project .
```

## Known Limitations

- Source refs verify file existence and optional hashes, but line-number refs and moved-file diagnostics are not yet complete.
- Audit catches missing sources, secret-like receipt content, and missing Lumi state, but contradiction and stale-package/deploy-fact checks are still future work.
- Backend adapters are not implemented yet. v0.2.0 proves the local contract shape before adapters.
- Recall returns a lead packet, but the recall broker needs stronger ranking, freshness, and contradiction handling.
- The local MCP facade is intentionally deferred until the CLI trust loop has more pilot evidence.
- npm publish requires a one-time password before `chainseal@0.2.0` can be live on the registry.

## Next Proof Target

v0.3 should prove stronger source verification:

- line-number refs;
- optional file hashes in normal workflows;
- clearer missing-source diagnostics;
- moved-file or renamed-file hints;
- fixture coverage for safe, poisoned, stale, and secret-like memories.

The proof target:

```text
Show that Chainseal can reject stale or unsupported memory claims without making normal source-backed memory writes painful.
```
